## A. 總驗收摘要

平台整體架構健全：後台全面強制登入（middleware JWT）、API 有角色驗證、薪資與異常管理已限主管、SQL 全部參數化無注入風險、公開連結（園所端、老師簡歷、報告）採 HMAC 簽章 token 且有效期限制、有每日備份與一致性檢查 cron。核心流程（園所→課程→排班→週課表→出勤→薪資／請款）是通的。

主要風險集中在四塊：

1. **LINE 發送無紀錄**：訊息發送失敗完全無痕，出事無法追查（最大單一風險）。
2. **備份不完整**：多張 raw SQL 表（請假、代課詢問、請款、異常、器材流向等）不在備份範圍，且沒有還原程序。
3. **請款／代課流程缺鎖**：請款可重複產生、代課確認非原子操作，並發操作可能產生重複或不一致資料。
4. **效能仍有 N+1**：/api/schools 已修（本次），但請款產生、LINE 週課表推播、報課查詢仍有逐筆查詢。

結論：**可以上線，但 C 節高風險 6 項建議上線前修完**，預估工程量約 3–5 個工作天。

## G. 資安修正建議

**已確認安全的部分**：後台全路徑強制登入；AUTH_SECRET 正式環境未設定會直接啟動失敗（不會用預設值）；requireRole 每次即時檢查帳號 isActive（停用帳號 token 立即失效）；薪資路徑限 owner 角色；異常管理已限主管（本次完成）；所有 $queryRawUnsafe 皆參數化，無 SQL injection；9 支 cron 全部驗 CRON_SECRET；LINE token 全部走環境變數，程式碼無硬編碼金鑰；公開 token 為 HMAC-SHA256 簽章 + timingSafeEqual + 有效期（簡歷 180 天、報告 90 天），無法被猜出；簡歷公開頁只回傳姓名與簡歷欄位，不含電話／薪資；照片上傳有型別與 2MB 限制。

**要修**：C-1 LINE 發送紀錄、C-5 審核操作寫 AuditLog、D-5 園所端事件遮罩、D-6 cron secret 移出 URL、D-8 欄位白名單、D-10 軟刪除。另建議：登入加失敗次數限制（防暴力破解）；老師簡歷 token 180 天偏長，可縮至 30 天（填完即失效更佳）。

## H. 流程順序改善建議

流程主幹（園所→課程→排班→課表→出勤→薪資/請款）順暢。斷點依優先序：

1. 請假核准→自動詢問代課（本次已做）→**代課確認後課表未強制同步**：確認代課時應自動建立/更新該日出勤的代課老師欄位，而不是等行政手動改。
2. 課後回報→課程進度是單向的：老師回報後進度需行政確認才更新，缺「超過 48 小時未確認」提醒。
3. 園所改時間（課程異動）核准後，原老師只收到通知、不需回覆確認 → 建議要求老師按「已知悉」。
4. 行政容易漏處理的三處：代課無人接（目前要自己去看）、回報未確認、請款月底未產生 → 都應進異常中心每日提醒。

## I. 資料一致性檢查建議

現有 consistency-check cron（每日 09:00）已檢查部分項目。建議擴充為 15 項完整清單：

1. 課程無指派老師 2. 課程園所缺地址 3. 老師缺電話或 LINE 未綁定 4. 已核准請假但代課狀態仍為空/searching 超過 48h 5. 有代課紀錄但當日出勤老師未更新 6. 有出勤但當月薪資單缺該筆 7. 課程有人數但當月無請款明細 8. 有課後回報但課程進度未更新（>7 天）9. LINE 已發送但對應狀態未變更（需先有發送紀錄表）10. 學期評量對不到有效課程 11. AI 報告缺學生或課程關聯 12. 證書園所名與實際上課園所不符 13. 簡歷不完整（缺照片/學歷/專長）但已開放園所查看 14. 園所人數異動已核准但請款未含新數 15. 同老師同時段重複排課。

每項檢查結果寫入 SystemAlert，主管在異常管理頁統一處理。

## K. 老師手機端優化建議

現況：老師無網頁帳號，全靠 LINE 選單＋公開 token 頁面——這個模式對老師反而簡單，建議維持。優化：1) LINE 選單固定四鍵：我的課表／打卡／回報／請假；2) 回報表單欄位減至必填 3 項內，30 秒可完成；3) 請假選課清單已擴至本月+下月 25 筆（本次完成）；4) 代課詢問卡片按鈕保持大按鈕兩選項；5) 公開頁按鈕高度 ≥44px、表格改卡片式；6) 錯誤訊息一律中文白話（已大致符合）。
